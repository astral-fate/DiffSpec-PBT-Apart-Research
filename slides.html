DiffSpec-PBT · The Problem
02 / 16
The Problem

Proofs are cheap. Specs are not.

LLMs can generate formal pre/post-condition specs. No-one can tell if a generated spec actually captures intent. Verify the wrong spec and you launder a bug into a theorem.

  • SPOTs / SMT-based validation works but requires decidability and a second spec — re-introduces the same bottleneck.
  • Feng & Sergey 2026 show PBT is unreasonably effective at validating human-written Lean specs.
  • Our move: use PBT on LLM-generated specs — and use a second LLM's spec as the oracle.
DiffSpec-PBT · Insight
03 / 16
Insight

Different LLMs disagree. PBT shows you where.

Prompt two independent spec authors with the same natural-language requirement. They produce different pre/post-conditions. Property-based testing surfaces a concrete input or output on which they disagree — a witness of intent drift. Works with any pair of OpenAI-compatible inference endpoints.

No SMT. No Lean. No golden reference. No manual proof effort.

DiffSpec-PBT · Architecture
04 / 16
Architecture

One requirement, two specs, three checks.

  • Provider-agnostic: OpenAI-compatible REST; swap models without changing the differ.
  • N-way eval: run every pair among N authors — drift on any pair flags the task.
  • Fixture mode by default — headline numbers reproduce with no API key.
  • Live mode regenerates fixtures via bash scripts/gen_fixtures.sh.
DiffSpec-PBT · The Three Checks
05 / 16
Three Checks

Each disagreement carries its own meaning.

① Soundness

Sample input i, compute o = impl(i). If post_A(i,o) ≠ post_B(i,o), the rejecting spec is OVER-specified.

② Uniqueness

Sample input i and mutated output o'. If post_A(i,o') ≠ post_B(i,o'), the accepting spec is UNDER-specified.

③ Precondition

Find i where pre_A(i) ≠ pre_B(i). The specs disagree on what counts as a valid input.

Each disagreement is auto-classified into one of five types: OVERSPEC, UNDERSPEC, PRECOND_MISMATCH, EDGE_DRIFT, CRASH.

DiffSpec-PBT · Why differential
06 / 16
Why differential

Cross-model diversity replaces the golden reference.

# Requirement: truncate a token sequence
# to at most N tokens.

# Stricter spec (prefix-only):
def post(i, o):
    tokens, n = i
    expected_len = min(n, len(tokens))
    return o == tokens[:expected_len]

# Weaker spec (any subsequence ≤ n):
def post(i, o):
    tokens, n = i
    return len(o) <= n and \
           all(x in tokens for x in o)

# PBT finds: input=([0], 1), o'=[]
# weaker accepts []; stricter rejects
# → UNDERSPEC in 46 examples
  • We never need the right spec. We just need two different wrong ones.
  • Different LLMs have different training-data blind spots, so they tend to make different mistakes.
  • Hypothesis shrinks each counterexample to a minimal witness — the diagnosis is auditable.
  • The only thing the methodology requires is that both specs be executable Python.

Limitation: correlated failure. If every author encodes the same wrong belief, the differ stays silent — run more providers and more pairs; mutation testing of the differ is future work.

Case Study — Real-World BRD

The methodology earns its keep.

Requirement (verbatim from Vamos Argentina BRD §III.8):

"If two or more fans have identical scores, the tie is broken by, in order:
(1) higher total Gameplay points,
(2) more unique platforms shared,
(3) earlier submission timestamp."

Three-tier tiebreaker. One model encodes all three rules. The other encodes only the primary total_score tier.

            (0, 0, 0, 1_700_000_000)]

            (0, 0, 0, 1_700_000_001)]
  strict: False (timestamp order wrong)
  weak  : True  (primary key matches, ok)

DiffSpec-PBT · Implementation
13 / 16
Implementation

1,500 lines of Python. No SMT. No Lean.

  • Language: Python 3.10+ — dataclasses, type hints.
  • PBT: hypothesis with shrinking.
  • LLM access: any OpenAI-compatible endpoint; our eval used four providers (NIM + Groq).
  • Tests: pytest + property-based tests on the differ itself.
  • Deps: hypothesis only at runtime. Live mode adds requests.
# Reproduce from a fresh clone:
$ git clone <repo>
$ cd DiffSpec-PBT-Apart-Research
$ pip install -e ".[test]"

$ make test     # unit + property-based tests
$ make eval     # 7/11 drift (any pair), ~22s

# Regenerate fixtures (provider keys in .env):
$ bash scripts/gen_fixtures.sh --task 06_token --model <alias>

DiffSpec-PBT · Limitations
14 / 16
What This Doesn't Solve

Owning the failure modes.

L1 · Correlated failure

If every author makes the same wrong choice, we stay silent. Two authors is the minimum; more providers and pairs help.

L2 · Executability

Both specs must run as Python. Non-decidable but executable specs are fine; specs needing a prover are out of scope.

L3 · Generation noise

Even at temperature=0.0, some providers return truncated or invalid Python. Fixture mode insulates the eval; live mode must retry.

L4 · Strategy authoring

Input strategies are hand-written. A weak strategy misses real disagreements. Future: LLM-authored strategies validated by the same differ.

L5 · No automatic repair

We surface the drift; we don't fix it. The taxonomy type points to a repair heuristic — agent-loop work for future iteration.

L6 · Pairwise engine

Each check compares two specs at a time; the harness already runs all author pairs. Majority-vote aggregation is future work.

DiffSpec-PBT · Future Work
15 / 16
Future Work

From prototype to production gate.

1 · Lean port

Re-implement on top of Lean's Plausible so validated specs flow directly into proof obligations.

2 · More providers + majority vote

Add Mistral, DeepSeek, Nemotron, etc. Aggregate pair-wise witnesses with majority vote; flag all-agree-and-wrong triples.

3 · Mutation testing the differ

Mutate the specs and verify the differ catches the mutation. Like mutation testing of test suites, applied to spec validators.

4 · Automatic spec repair

Each taxonomy type points to a repair heuristic (UNDERSPEC → add a conjunct; OVERSPEC → weaken one). Agent-loop closes the gap.

5 · CI integration

Drop-in GitHub Action that runs DiffSpec-PBT on every PR modifying a spec. Sub-second per task means it fits the per-PR budget.

6 · LLM-authored strategies

Use an LLM to generate the input/output Hypothesis strategy, validated by the differ in turn. Closes the bootstrap loop.

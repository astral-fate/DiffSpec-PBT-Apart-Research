## Problem

Specifications are the bottleneck in trustworthy AI software (Dodds 2025; Dougherty 2026). LLMs can generate them but no one can tell if a generated spec correctly captures intent. Symbolic proof of spec correctness (SPOTs, Lean) is expensive. Feng & Sergey (2026) show property-based testing is unreasonably effective at validating *human-written* Lean specs. We push further.

## Idea

**Prompt two different LLMs to independently generate formal specifications for the same requirement. Use property-based testing to find inputs/outputs where the two specs disagree.** Each disagreement is a concrete counterexample of intent drift, classified by type.

Cross-model diversity is the oracle. We never need to know the "right" spec вЂ” we exploit the fact that two LLMs with different training data tend to make *different* mistakes, and PBT surfaces the gap.

## Three differ checks

1. **Soundness** вЂ” generate `i`, compute `o = impl(i)`. If `post_A(i,o) в‰  post_B(i,o)` в†’ one spec is over-specified (rejects valid output).
2. **Uniqueness** вЂ” generate `i` and a *mutated* `o'`. If `post_A(i,o') в‰  post_B(i,o')` в†’ one spec is under-specified (accepts invalid output).
3. **Precondition mismatch** вЂ” find `i` where `pre_A(i) в‰  pre_B(i)` в†’ disagreement on the valid input domain.

## Disagreement taxonomy

| Type | Trigger |
|---|---|
| `OVERSPEC` | Soundness fail вЂ” implementation output accepted by one spec, rejected by the other |
| `UNDERSPEC` | Uniqueness fail вЂ” a clearly-wrong mutated output accepted by one spec, rejected by the other |
| `PRECOND_MISMATCH` | Specs disagree on what's a valid input (edge-case handling, type assumptions) |
| `EDGE_DRIFT` | Disagreement only on boundary inputs (empty, None, max-size) вЂ” methodology bias toward edge cases |
| `CRASH` | One spec crashes/raises on an input the other handles |

| Module | Role |

## Benchmark вЂ” 10 tasks

**Algorithmic (5):** sort, dedup, binary_search, kth_smallest, merge_sorted.
**Security-relevant (5):** tokenizer_truncate, prompt_priority, permission_check, rate_limit, path_sanitize.

Each task ships as:
```
benchmark/<NN>_<name>/
в”њв”Ђв”Ђ requirement.md       # natural-language requirement
в”њв”Ђв”Ђ impl.py              # reference implementation (the "code under spec")
в”њв”Ђв”Ђ strategy.py          # Hypothesis strategies for inputs + mutated outputs
в””в”Ђв”Ђ fixtures/
    в”њв”Ђв”Ђ claude.py        # spec as generated by Claude (or a realistic asymmetric stand-in)
    в””в”Ђв”Ђ gpt.py           # spec as generated by GPT-4o (or a realistic asymmetric stand-in)
```

Fixtures are **hand-crafted to be realistic**: each pair reflects an asymmetry a real LLM pair would plausibly produce, and engineered so the differ surfaces at least one genuine disagreement classified into the taxonomy.

## LLM mode

**Fixture mode (default).** Ships 20 specs (10 tasks Г— 2 models). Eval reads these files. Reproduces on a fresh clone with no API keys. *This is the headline-number-reproducibility mode.*

**Live mode (`--live`).** If `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` are set, calls real Claude + GPT and caches results into `benchmark/<task>/cache/`. Used for demo + regenerating fixtures.

Why fixture-first: judges shouldn't need paid API keys to reproduce a paper claim. The methodology is exactly the same; only the spec-generation step changes provider.

## Non-interference with existing related work

| Prior work | What it does | What we do differently |
|---|---|---|
| Feng & Sergey 2026 | PBT on *human-written* Lean specs | PBT on *LLM-generated* specs, differentially |
| Rao DiffSpec 2024 | Differential testing of LLM-generated *code* | Differential testing of LLM-generated *specs* |
| SPOTs (Lahiri 2026) | SMT-based small proofs of spec properties | PBT вЂ” no SMT needed, faster, scales |
| AutoSpec/SpecGen | LLM spec generation, no validation | We *validate* specs after generation |
